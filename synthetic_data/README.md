

Each batch starts from one real "seed" entry already in the dataset and produces several variations of it. The point is for the model to learn the *underlying pattern* rather than memorise one specific code sample, so every variation keeps the same security property but changes everything on the surface: the cover story (a login filter, an API gateway, a save-game file, and so on), how the code is organised (a class, a pair of functions, a static utility, a Spring component, ...), the exact coding idiom, and the secret key and message. When a seed comes from a real CVE, the actual published source snippet is copied into every variation **byte-for-byte**, so each entry stays a faithful example of the real bug or the real fix instead of invented code; only the surrounding scenario and a small self-contained reproduction are rewritten.



Before an entry is kept, it has to pass three kinds of check. First, **static checks** confirm the displayed code actually contains the construct it is supposed to -- the AES-CBC padding check for vulnerable entries, the GCM or HMAC machinery for safe ones -- and that the surrounding prose stays neutral and never gives away the answer. Second, a **smoke test** actually runs the code: for a vulnerable entry the project's reference exploit must really recover the plaintext, and for a safe entry that same exploit must **fail**. That "inverted" test is how we prove a fix genuinely closes the oracle rather than just looking like it does. Third, a **similarity check** makes sure the variations are genuinely different from one another (and from the seed): we measure text similarity between every pair of prompts and require it to stay below 0.85. For the real-CVE batches a large part of each prompt is the shared verbatim snippet, which would inflate a raw comparison, so there we measure similarity on the *authored* portion only (with the shared snippet removed) and apply the limit to that.
