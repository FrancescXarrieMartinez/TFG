# Apache MyFaces batch (5 safe variations)

This batch contains five synthetic **NOT_VULNERABLE** entries derived from the seed at `dataset.json` index 10 (`cve-2010-2057-safe`), the Apache MyFaces fix for CVE-2010-2057. Here the cipher stays AES-CBC, but an **HMAC tag is added and verified before decryption** (encrypt-then-MAC): a tampered message fails the MAC check and is thrown out before any padding code runs, so there is no oracle. All five entries are in the main dataset.

This is a safe batch, so it follows the inverted methodology: the answer is "not vulnerable", `expected_plaintext` is null, and the vulnerable-side exploit is required to **fail** against the adapter -- which it did for all five. The difference from the Shiro safe batch is that AES-CBC is now *required* (the safety comes from the HMAC layer, not from changing the cipher), so the static check looks for the HMAC machinery and confirms the MAC comparison comes *before* the decryption step. The real MyFaces `StateUtils` source is copied verbatim into every entry.

The five variations differ in scenario (an enterprise JSF app, an internal CMS, an auditable form system, a banking portal, and a microservice facade) and in code structure. Because the verbatim MyFaces snippet is large -- about three-quarters of each prompt is shared, identical text -- a raw similarity comparison is dominated by that shared block, so the diversity check is applied only to the authored portion; there the most similar pair scored 0.75, under the 0.85 limit. That figure is a little higher than the other batches because encrypt-then-MAC has essentially one correct shape and several entries call into the same library, but the code structures are still distinct.
